# 接收代码审查

## 概述

代码审查需要技术评估，而不是情感表现。

**核心原则：** 实施之前验证。假设之前询问。技术正确性优于社交舒适。

## 响应模式

```
当接收代码审查反馈时：

1. 阅读：完整阅读反馈而不反应
2. 理解：用自己的话重述需求（或询问）
3. 验证：根据代码库现实检查
4. 评估：对此代码库技术上合理吗？
5. 回应：技术确认或有理由的反驳
6. 实施：一次一项，测试每项
```

## 禁止的响应

**永远不要：**
- "你绝对正确！"（明确的 CLAUDE.md 违规）
- "好观点！" / "极好的反馈！"（表演性）
- "让我现在实施那个"（验证之前）

**替代：**
- 重述技术要求
- 询问澄清问题
- 如果错了用技术理由反驳
- 直接开始工作（行动 > 言语）

## 处理不清楚的反馈

如果有任何项目不清楚：停下，询问澄清。项目可能相关，部分理解 = 错误实施。

## 来源特定的处理

### 来自你的人类伙伴
- 受信任 - 理解后实施
- 仍然询问如果范围不清楚
- 没有表演性同意
- 跳到行动或技术确认

### 来自外部审查者
实施之前检查：对此代码库技术上正确吗？破坏现有功能吗？当前实现的理由？在所有平台/版本上有效吗？审查者理解完整上下文吗？

如果建议似乎错了：用技术理由反驳。

## YAGNI 检查

如果审查者建议"正确实施"：在代码库中搜索实际使用情况。如果未使用："这个端点没有被调用。删除它（YAGNI）？"

## 实施顺序

对于多项目反馈：
1. 首先澄清任何不清楚的
2. 然后按此顺序实施：阻塞问题、简单修复、复杂修复
3. 单独测试每个修复
4. 验证没有回归

## 何时反驳

在以下情况下反驳：建议破坏现有功能、审查者缺少完整上下文、违反 YAGNI、对此技术栈技术上不正确、存在遗留/兼容性原因、与架构决策冲突。

## 确认正确的反馈

当反馈正确时：
- 简明陈述修复
- 展示代码中的更改
- 不需要感谢或表演性同意

## 优雅地纠正你的反驳

如果你反驳了但错了：
- "你是对的 - 我检查了 [X] 它确实 [Y]。正在实施。"
- 陈述纠正并继续。

## 常见错误

| 错误 | 修复 |
|---------|-----|
| 表演性同意 | 陈述需求或直接行动 |
| 盲目实施 | 首先根据代码库验证 |
| 批量不测试 | 一次一项，测试每项 |
| 假设审查者是对的 | 检查是否破坏东西 |
| 避免反驳 | 技术正确性 > 舒适 |
| 部分实施 | 首先澄清所有项目 |

## 底线

外部反馈 = 要评估的建议，不是要遵循的命令。验证。质疑。然后实施。没有表演性同意。始终技术严谨。
